# Wandrix Agent Rules

These rules apply to the whole repository.

## Product Shape

Wandrix is a conversation-first AI travel planner.

The core product experience is:
- left side: AI conversation workspace
- right side: live trip board
- final state: a polished brochure-style travel output

Every major implementation decision should preserve that shape.

Planning happens primarily in `/chat`.
Flights, hotels, activities, and similar module routes are supporting views for inspection, summaries, and later exports, not the main place where trip decisions are made.

## Core Architecture Rules

1. The frontend talks to FastAPI for product data.
Do not let the frontend write trip data directly to the database.

2. Supabase is split by responsibility.
- Supabase Auth is used for authentication.
- Supabase Postgres is used for persistence.
- FastAPI remains the application backend.

3. LangGraph is embedded inside the backend.
Do not introduce a separate graph service unless there is a clear operational reason.

4. One browser session can contain many trips.
Each user-initiated conversation is one trip.

5. One trip owns one live board and one LangGraph thread.
Keep `trip_id` and `thread_id` tightly associated.

6. Chat is not the source of truth for the trip board.
The structured trip draft is the source of truth for the live board and brochure.

## Backend Rules

1. Routes stay thin.
HTTP concerns belong in routes; orchestration belongs in services.

2. Provider-specific logic stays behind integrations and adapters.
Do not leak Amadeus, Geoapify, Mapbox, Ticketmaster, or provider-specific payloads into route files or frontend contracts.

3. Persist product state explicitly.
Browser sessions, trips, and trip drafts must be saved in owned tables.

4. Normalize external provider data before using it in the product.
The rest of the app should work with internal models, not raw provider responses.

5. Keep AI outputs validated.
Anything generated by the model must be validated before it becomes structured trip data.

6. Alembic only manages app-owned tables.
Never let migrations try to modify unrelated Supabase tables.

7. Do not use deterministic extraction for planner understanding.
Trip understanding must stay LLM-first, schema-validated, and ambiguity-aware rather than falling back to regex or keyword parsing.

8. Prefer clarification over silent locking when user intent is ambiguous.
If the planner is not confident about a field, it should ask or mark it as inferred instead of hard-committing a guessed value.

## Frontend Rules

1. The product is conversation-first, not dashboard-first.
Avoid generic admin/dashboard layouts and keep the UI centered on trip planning.

2. Keep the live trip board structured.
The board should reflect persisted trip draft data, not only chat text.

3. Use shared tokens for fonts, colors, and visual system choices.
Do not hardcode random colors and font stacks inside feature components.

4. Prefer global visual tokens.
Theme values should come from `globals.css` and shared styling tokens before adding one-off values.

5. Do not scatter API calls.
Use shared API helpers and typed contracts.

6. Keep the final brochure HTML-first.
Build a high-quality brochure page before investing in PDF export tooling.

## Styling Rules

1. Colors must come from global theme tokens.
If a new color is needed, add it intentionally to global tokens instead of hardcoding it in a component.

2. Fonts must come from global layout/theme configuration.
Do not introduce one-off font choices inside feature components.

3. Visual changes should preserve a premium travel-product feel.
Avoid default-looking SaaS UI, generic purple gradients, or filler dashboard patterns.

## Change Log Rules

1. Every meaningful change must be appended to `CHANGELOG.md`.

2. Each changelog entry must contain:
- date
- short title
- technical summary
- plain-English summary
- files or areas touched

3. The plain-English summary should explain the change for a non-technical stakeholder.

4. Do not replace old entries.
Append new entries in reverse-chronological order.

## Documentation Rules

1. Keep architecture docs current when major decisions change.

2. If the stack or ownership model changes, update:
- `docs/architecture.md`
- `docs/decision-log.md`
- relevant coding rules

3. Do not silently change core patterns.
If the project moves away from a documented pattern, record that decision.

## Implementation Discipline

1. Infrastructure setup should be separated from feature logic when possible.

2. Before implementing a major feature, confirm the supporting contracts exist:
- auth
- persistence
- schema
- API contract
- UI ownership

3. Do not skip adapters and go straight from providers to UI.

4. Do not bypass typed models for speed.

5. Do not reintroduce heuristic parsing into the planner.
If planner intelligence needs improvement, strengthen the structured LLM flow and clarification behavior instead of adding brittle parsing rules.

6. Do not let files grow without bound.
If a file starts becoming too large, split it by responsibility instead of continuing to append more logic to the same module.

## Testing and Verification

1. Backend changes should at minimum pass import/compile checks.

2. Frontend changes should at minimum pass lint and build.

3. Schema, migration, and persistence changes should be validated against the real database flow.

## Priority Order

When unsure what to build next, prefer this order:

1. persistence and contracts
2. auth and ownership
3. conversation runtime
4. live trip board state
5. brochure rendering
6. deeper travel intelligence
